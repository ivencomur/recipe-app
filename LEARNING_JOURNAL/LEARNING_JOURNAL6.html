4) Problems I Ran Into & Solutions

Python Version Mismatch Problem

When I tried to install mysql-connector-python, I got a "module not found" error even though pip said it installed successfully. Turns out I had TWO Python versions on my computer (3.9 and 3.13), and python pointed to 3.13 while pip pointed to 3.9!
Solution: Used py -3.9 to explicitly run Python 3.9 for everything. This made sure both the installation AND the script execution used the same Python version. Lesson learned: Always check which python and which pip match!

The SQL Injection Vulnerability (BIG DEAL!)

My original search function used an f-string to build the SQL query like this:
sql = f"SELECT * FROM Recipes WHERE ingredients LIKE '%{search_ingredient}%'"
When I got the code checked, I learned this was DANGEROUS! If someone typed something malicious like '; DROP TABLE Recipes; --, they could delete my entire database!
Solution: Changed to parameterized queries:
sql = "SELECT * FROM Recipes WHERE ingredients LIKE %s"
cursor.execute(sql, ('%' + search_ingredient + '%',))
The %s placeholders protect against SQL injection because MySQL escapes the input properly. This is CRITICAL for any real application!

No Error Handling for User Input

My original code just used int(input(...)) without checking. If someone typed "ten" instead of 10, the program crashed with a ValueError.
Solution: Wrapped ALL numeric input in try-except blocks:
try:
  cooking_time = int(input("..."))
except ValueError:
  print("Error: Please enter a valid number.")
  return
Now the program handles mistakes gracefully instead of crashing. Users get helpful messages and can try again.

MySQL Doesn't Understand Python Lists

I wanted to store my ingredients as a Python list like ["Flour", "Sugar", "Eggs"], but MySQL's VARCHAR column can only store text strings.
Solution: Convert between formats:
Storing: ingredients_str = ", ".join(ingredients) → "Flour, Sugar, Eggs"
Retrieving: ingredients_list = ingredients_str.split(", ") → ["Flour", "Sugar", "Eggs"]
This conversion happens in multiple places (create, search, update), so I had to remember to do it consistently everywhere.

Difficulty Needs Recalculation on Updates

When I update a recipe's cooking time OR ingredients, the difficulty might change. But difficulty depends on BOTH values, so I need the other one to recalculate.
Solution: When updating cooking_time, I retrieve ingredients from the database first. When updating ingredients, I retrieve cooking_time first. Then I recalculate difficulty and update BOTH the changed field AND difficulty in the same operation. This was trickier than I expected!

MySQL Shell Was in JavaScript Mode

When I tried to create the cf-python user, MySQL Shell kept giving syntax errors. The prompt said "MySQL JS >" which I didn't understand at first.
Solution: Typed \sql to switch to SQL mode. The prompt changed to "MySQL SQL >" and THEN my commands worked. Then I connected with \connect root@localhost and entered my password. Lesson: MySQL Shell has different modes - make sure you're in the right one!

Redundant Commit in Main Menu

My original code called conn.commit() in the main_menu function at the end. But each CRUD function (create, update, delete) already commits its own changes, so this was unnecessary!
Solution: Removed the redundant conn.commit() and just kept conn.close(). Each function is responsible for committing its own transaction. The main menu just closes the connection when exiting.

6) How The Code Works

Program Startup

1. Import mysql.connector 2. Connect to MySQL server (host='localhost', user='cf-python') 3. Create cursor for executing commands 4. Create database 'task_database' if it doesn't exist 5. Switch to using task_database 6. Create Recipes table if it doesn't exist 7. Call main_menu() to start the application

The Main Menu Loop

while True: Display menu options (1-5) Get user's choice if choice == "1": Call create_recipe() elif choice == "2": Call search_recipe() elif choice == "3": Call update_recipe() elif choice == "4": Call delete_recipe() elif choice == "5": Break out of loop (exit) else: Show "invalid choice" message After loop ends: Close database connection Program terminates

Creating a Recipe (Flow)

1. Ask user for recipe name (string input) 2. Try to get cooking time (integer) - If ValueError: Show error, exit function 3. Try to get number of ingredients (integer) - If ValueError: Show error, exit function 4. Loop to collect each ingredient name 5. Calculate difficulty based on time and ingredient count 6. Convert ingredient list to comma-separated string 7. Build parameterized INSERT query 8. Execute query with values 9. Commit changes to database 10. Show success message

Searching by Ingredient (Flow)

1. SELECT all ingredients from Recipes table 2. Split comma-separated strings into individual ingredients 3. Build list of unique ingredients (no duplicates) 4. Display numbered list to user 5. Try to get user's ingredient choice (integer) - If ValueError: Show error, exit - If IndexError: Show error, exit 6. Use choice to get ingredient name from list 7. Build parameterized SELECT query with LIKE operator 8. Execute query (finds recipes containing that ingredient) 9. Loop through results and display each recipe

Updating a Recipe (Flow)

1. SELECT all recipes and display them with IDs 2. Try to get recipe ID to update (integer) - If ValueError: Show error, exit 3. Ask which field to update (name, time, or ingredients) 4. If updating name: - Get new name - UPDATE name WHERE id = recipe_id 5. If updating cooking_time: - Get new cooking time - SELECT ingredients for this recipe - Recalculate difficulty - UPDATE both cooking_time AND difficulty 6. If updating ingredients: - Get new ingredients list - SELECT cooking_time for this recipe - Recalculate difficulty - UPDATE both ingredients AND difficulty 7. Commit changes 8. Show success message

Key Security Pattern

UNSAFE (SQL Injection vulnerable): sql = f"SELECT * FROM Recipes WHERE id = {user_id}" SAFE (Parameterized query): sql = "SELECT * FROM Recipes WHERE id = %s" cursor.execute(sql, (user_id,)) The %s placeholder tells MySQL "this is data, not code" MySQL properly escapes the value to prevent injection

7) What I'll Remember Most

Technical Takeaways

  • Databases are WAY more powerful than binary files for storing data permanently
  • SQL injection is a REAL security threat - never concatenate user input into SQL strings
  • Parameterized queries using %s are the safe way to include user data in SQL
  • Always validate user input with try-except blocks - programs should never crash on bad input
  • MySQL doesn't support Python lists - must convert to/from comma-separated strings
  • Connection and cursor are different - connection is the session, cursor executes commands
  • Changes aren't permanent until you call commit()
  • AUTO_INCREMENT is super convenient - the database assigns IDs automatically

Process Lessons

  • Getting code reviewed revealed security problems I didn't know existed
  • Python version mismatches cause confusing "module not found" errors
  • MySQL Shell has different modes - make sure you're in SQL mode for SQL commands
  • When things depend on each other (like difficulty depends on time and ingredients), updates get complex
  • Error handling isn't optional - it's what makes programs professional vs amateur
  • Building step-by-step with testing at each stage catches problems early

Big Picture Understanding

  • Real applications use databases: Every website, app, and professional software stores data in databases like MySQL
  • Security matters from day one: SQL injection is ONE example of how bad code can be exploited - there are many others
  • Data modeling is crucial: Choosing the right data types and structure makes everything easier later
  • CRUD is fundamental: Create, Read, Update, Delete - these four operations power almost every application
  • Professional code handles errors: Try-except blocks aren't extra - they're required for production code

The Memorable Moments

  • Understanding that placeholders aren't just for formatting - they're for SECURITY
  • Realizing that commit() is like saving a file - changes exist but aren't permanent until you save
  • Discovering, or at least trying to see how SQL LIKE with wildcards works - it's pattern matching, not exact matching
  • Understanding why we convert lists to strings - it's a limitation of SQL, not bad design
  • Recognizing that every function needs its own error handling - can't just handle errors at the top level
