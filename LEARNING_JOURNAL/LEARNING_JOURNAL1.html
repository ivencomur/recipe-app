4) Challenges & Solutions

Python version differences

Course specified Python 3.8.7; I used 3.9.x (within acceptable range).
Solution: Explicitly used py -3.9 launcher to ensure consistent version across all venv operations.

Tool compatibility

Course referenced virtualenvwrapper-win, which requires Command Prompt (incompatible with Git Bash).
Solution: Used built-in venv module‚Äîsimpler, cross-platform, no additional installation needed.

Path handling in Git Bash

Spaces in directory paths caused navigation issues; quotes around ~ prevented tilde expansion.
Solution: Used ~/path\ with\ spaces format or absolute paths with escaped spaces: /c/Users/Ivan/My\ Folder.

Command syntax ambiguity

Course materials showed > pip freeze > requirements.txt which was confusing.
Solution: Learned > is shell redirection (not part of pip). Correct syntax: pip freeze > requirements.txt.

Command variations

Uncertain whether to use python, python3, py, pip, or pip3.
Solution: Within activated venv, use python and pip. Outside venv, use py -3.9 to target specific version.

9) Key Takeaways

Technical insights:

  • Virtual environments are essential for dependency management
  • Built-in venv module is simpler and more portable than third-party alternatives
  • requirements.txt makes projects reproducible across systems
  • Python commands vary by platform‚Äîtest early and document what works

Process lessons:

  • Environment setup often takes longer than estimated
  • Course materials may reference tools that have evolved or been replaced
  • Multiple valid approaches exist for the same goal (virtualenvwrapper vs. venv)
  • Systematic troubleshooting (isolate one variable at a time) is more effective than trial-and-error

Professional practices:

  • Documentation serves both current reviewers and future self
  • Screenshots provide proof of work and aid in debugging
  • Seeking help (documentation, forums, AI) is standard practice in development
  • Understanding why tools work matters more than memorizing how
